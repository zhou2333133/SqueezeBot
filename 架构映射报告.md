## 一、模块分层归属

### 第 1 层：妖币扫描 = 选币层

| 模块 | 归属 | 说明 |
|---|---|---|
| `scanner/` (candidates, ai_gateway, yaobi_scanner) | ✅ 第 1 层 | 发现候选币，评分，输出候选列表 |
| `scanner/knowledge_base.py` | ✅ 第 1 层 | 记录 AI 判断的反馈，不做交易决策 |

**职责**: 发现候选交易对，不直接决定下单。

---

### 第 2 层：AI 判断 = 候选解释层

| 模块 | 归属 | 说明 |
|---|---|---|
| `scanner/ai_gateway.call_single_coin_judge()` | ✅ 第 2 层 | LLM 解释币种关注理由，不下单、不改参数 |
| `scanner/single_coin_judge.py` (deprecated) | ✅ 第 2 层 | 已废弃，功能被 evolver 替代 |

**职责**: 用人话解释为什么关注这个币，不直接下单、不改参数。

---

### 第 3 层：规则学习器 = 规则适配层 ← **当前新增**

| 模块 | 归属 | 说明 |
|---|---|---|
| `market_state.py` | ✅ **第 3 层** | 纯规则判断市场阶段，不拦截交易、不改参数 |
| `trade_context.py` | ✅ **第 3 层** | 提取入场特征画像，记录到 trade dict，不拦截 |
| `learning_memory.py` | ✅ **第 3 层** | 按 symbol×state_key×rule_id 存储历史表现，不拦截 |
| `rule_selector.py` | ✅ **第 3 层** | 软判断：冷启动保护 + 弱过滤 + 历史低分 block |

**职责**: 判断当前币种+市场状态适合哪套规则，是**软判断层**，不是硬风控。

---

### 第 4 层：Evolver = 参数进化层

| 模块 | 归属 | 说明 |
|---|---|---|
| `strategy_evolver.py` | ✅ 第 4 层 | 生成 proposal，不直接写 config |
| `evolver_runtime.py` | ✅ 第 4 层 | 运行调度、冻结恢复，不写 config |
| `param_attribution.py` | ✅ 第 4 层 | 因果回溯，不写 config |

**职责**: 根据历史表现产生参数/权重修改 proposal，必须经过 risk_guard 审核才能写入。

---

### 第 5 层：risk_guard = 硬风控层

| 模块 | 归属 | 说明 |
|---|---|---|
| `risk_guard.py` | ✅ 第 5 层 | 硬风控：金额、仓位、杠杆、亏损、回撤、locked params |

**职责**: 执行前最后一道硬闸。只做"允许/拒绝"的二元判断，不做策略选择。

---

### 第 6 层：execution_router = 执行层

| 模块 | 归属 | 说明 |
|---|---|---|
| `execution_router.py` | ✅ 第 6 层 | 根据 PAPER/LIVE 选择执行器，返回统一结果 |
| `bot_scalp._exec_open()` | ✅ 第 6 层 | 委托 execution_router.execute()，不夹带策略判断 |

**职责**: 只负责执行，不负责判断行情。PAPER/LIVE 差别集中在此。

---

### 跨层辅助

| 模块 | 归属 | 说明 |
|---|---|---|
| `ai_reporter.py` | ✅ **跨层辅助** | 读取第 3/4 层数据生成中文日报，不参与交易决策 |
| `strategy_stats.py` | ✅ **跨层辅助** | 统计账本，不参与交易决策 |
| `data/strategy_trades.jsonl` | ✅ **跨层账本** | 唯一成交账本，所有层都读，仅第 6 层写入 |

## 二、每一层的输入/输出/权限

### 第 1 层：妖币扫描

| 项目 | 说明 |
|---|---|
| 输入 | 交易所公开数据（K线、OI、成交量）、链上数据、社交数据 |
| 输出 | `candidates_queue` → 候选币列表（含评分/标签） |
| 输出给 | 第 2 层 (AI 解释) + 第 3 层 (规则适配) + bot_scalp._detect_signal_v3 |
| 拦截交易 | ❌ 不允许 |
| 修改参数 | ❌ 不允许 |
| 修改用户边界 | ❌ 不允许 |
| 调用交易所 API | ✅ 只读（行情查询） |

### 第 2 层：AI 判断

| 项目 | 说明 |
|---|---|
| 输入 | 候选币数据 + MiniMax LLM |
| 输出 | `yaobi_decision_action`, `trade_permission` → 第 3 层 |
| 输出给 | 第 3 层 (规则适配) |
| 拦截交易 | ❌ 不允许（AI 判断只是解释，block 由第 3/5 层决定） |
| 修改参数 | ❌ 不允许 |
| 修改用户边界 | ❌ 不允许 |
| 调用交易所 API | ❌ 不允许 |

### 第 3 层：规则学习器（当前新增）

| 项目 | 说明 |
|---|---|
| **market_state** 输入 | K线缓存 (`kline_buffer`)、OI、Taker、Volume |
| **market_state** 输出 | `{trend, volatility, liquidity, momentum, confidence, state_key}` → 第 3 层内部 |
| **trade_context** 输入 | entry_context → 提取结构化特征 |
| **trade_context** 输出 | `_features` → 写入 trade dict → 第 3 层 learning_memory |
| **learning_memory** 输入 | trade dict (含 _state_key + _signal_type) |
| **learning_memory** 输出 | `get_rule_stats()` → 第 3 层 rule_selector |
| **rule_selector** 输入 | symbol + state_key + rule_id → learning_memory 查询 |
| **rule_selector** 输出 | `{allow, reason, confidence, memory_score, suggested_action}` → 第 5 层 |
| 输出给 | 第 5 层 risk_guard |
| **拦截交易** | ✅ **允许（软判断）** — 但冷启动保护确保样本不足时不拦截 |
| 修改参数 | ❌ 不允许 |
| 修改用户边界 | ❌ 不允许 |
| 调用交易所 API | ❌ 不允许 |

### 第 4 层：Evolver

| 项目 | 说明 |
|---|---|
| 输入 | `strategy_trades.jsonl`、`shadow_trades.jsonl`、`learning_memory.json`（通过 `get_weak_symbols()` 等） |
| 输出 | proposal list → 第 5 层 risk_guard.apply_proposals() |
| 输出给 | 第 5 层 |
| 拦截交易 | ❌ 不允许（Evolver 不直接参与交易） |
| 修改参数 | ⚠️ **仅通过 proposal** → 必须经第 5 层 risk_guard 审核 |
| 修改用户边界 | ❌ 不允许（LOCKED_PARAMS + boundary_config 双重保护） |
| 调用交易所 API | ❌ 不允许 |

### 第 5 层：risk_guard

| 项目 | 说明 |
|---|---|
| 输入 | 开仓信息 (symbol, amount, leverage, positions) + boundary_config |
| 输出 | `{allow: bool, reason: str}` → 第 6 层 |
| 输出给 | 第 6 层 execution_router |
| 拦截交易 | ✅ **允许（硬拦截）** — 执行前最后一道闸 |
| 修改参数 | ⚠️ **仅 apply_proposals()** — 审核 Evolver 的 proposal 后写入 |
| 修改用户边界 | ✅ 写入 boundary_config（但 AI/Evolver 不能调用） |
| 调用交易所 API | ❌ 不允许 |

### 第 6 层：execution_router

| 拦截交易 | ❌ 不允许（第 5 层已拦截完） |
| 修改参数 | ❌ 不允许 |

## 四、正确的数据流

```
yaobi_scanner / signal_engine (第 1 层)
  → 候选币列表
  →
  bot_scalp._detect_signal_v3 (第 1→3 层桥梁)
  → 检测到信号
  →
  market_state.classify() (第 3 层 — 新增)
  → 输出 state_key
  →
  trade_context.extract_features() (第 3 层)
  → 输出 _features (写到 trade dict)
  →
  rule_selector.should_trade() (第 3 层 — 新增)
  → 软判断：allow / weak_block / block
  → 输出: {allow, reason, confidence, memory_score}
  →
  risk_guard.check_open_order() (第 5 层)
  → 硬风控：金额/仓位/杠杆/亏损/回撤
  → 输出: {allow, reason}
  →
  execution_router.execute() (第 6 层)
  → PAPER：虚拟成交 | LIVE：trader API
  → 输出: 统一 _ExecResult
  →
  strategy_stats.record_trade() (跨层账本)
  → 写入 strategy_trades.jsonl
  →
  learning_memory.record_trade() (第 3 层)
  → 按 symbol × state_key × rule_id 更新记忆
  → 写入 learning_memory.json
  →
  param_attribution.feed_trade() (第 4 层)
  → 因果回溯
  →
  strategy_evolver.run_evolution_once() (第 4 层)
  → 读取 strategy_trades.jsonl + learning_memory.json
  → 产生 proposal
  →
  risk_guard.apply_proposals() (第 5 层)
  → 审核 proposal
  → 写入 settings.json (策略参数)
  →
  ai_reporter.generate_report() (跨层辅助)
  → 读取 learning_memory + strategy_stats + param_patches
  → MiniMax 生成中文日报
```

## 五、是否需要调整当前实现计划

### 结论：无需调整。模块归属正确。

| 模块 | 层 | 当前状态 | 是否需调整 |
|---|---|---|---|
| market_state.py | 第 3 层 | 新建，纯规则 | ✅ 正确 |
| trade_context.py | 第 3 层 | 已有，增强中 | ✅ 正确 |
| learning_memory.py | 第 3 层 | 已有，schema 升级 | ✅ 正确 |
| rule_selector.py | 第 3 层 | 新建，软判断 | ✅ 正确 |
| ai_reporter.py | 跨层辅助 | 已有 | ✅ 正确 |
| execution_router.py | 第 6 层 | 已有 | ✅ 正确 |
| risk_guard.py | 第 5 层 | 已有 | ✅ 正确 |

### 唯一需要注意的边界

`rule_selector` 的 `rank` 或 `suggested_action` 当前只输出 `allow/warn/weak_block/block`。如果未来需要输出"建议降低权重"或"建议减少仓位"，**必须**走 proposal 系统（第 4 层 → 第 5 层），不能直接从 rule_selector 改参数。当前实现符合此约束。
